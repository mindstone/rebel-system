Key Mental Models

Four core concepts for understanding Rebel's memory system:

Mental Model #1: The Two-Tier System

Think of Rebel's memory like a library with two sections:

  • Auto-loaded (README.md): Essential facts always present — your identity, preferences, current focus (50%+ utility)
  • On-demand (memory/topics/): Detailed context loaded when needed — project details, client history, workflows (<50% utility)

Topic files often point to external sources (Gmail, Notion, files) rather than duplicating content.

Why this matters: Keeps context manageable while ensuring essential facts are always available and detailed context is just a search away.

Mental Model #2: Topics vs Sources

Memory has two parallel systems:

  • Topics (memory/topics/): Curated knowledge — synthesized facts with source citations
  • Sources (memory/sources/): First-party captures — meetings, emails, Slack threads preserved with provenance

Key rule: Topics cite sources for provenance, but sources don't backlink to topics. This keeps sources stable when you reorganize topics.

Why this matters: You can trace any fact back to its original source, and sources stay clean even as your knowledge evolves.

Mental Model #3: Pointers, Not Copies

Don't think of memory files as storing information. Instead, think of them as a library catalog system:

  • ❌ Not this: Copy email content into memory files
  • ✅ This instead: Store Gmail query strings that retrieve current emails

The SSOT Principle: Every fact lives in ONE canonical place (Single Source Of Truth). Memory files point to that place, they don't duplicate it.

Why this matters: Prevents information from going stale, reduces duplication, and keeps memory files lightweight.

Mental Model #4: Two Privacy Zones

Think of your Rebel workspace as having two main privacy zones:

  1. Chief-of-Staff/ 🔒 - Your private space (never shared with anyone)
  2. work/Company/ 👥 - Shared work space (visible to your team)

Simple rule: Personal and sensitive information goes in Chief-of-Staff. Work content that your team should see goes in the work space.

Why this matters: Keeps decision-making simple — ask yourself "should my team see this?" and you'll know where to save it.

Overview: What is Rebel Memory?

Rebel's memory system is a structured knowledge architecture that helps the AI assistant maintain context about you, your work, and your preferences across conversations. Unlike simple chat history, it's organized into tiers, respects privacy boundaries, and uses pointers to avoid duplicating information.

The core problem it solves: You don't want to repeat yourself in every conversation ("I work at Company X", "My timezone is Y", "Here's how I prefer emails drafted"). But you also don't want everything loaded into every conversation (too much context), and you definitely don't want sensitive personal information accidentally shared in company spaces.

The solution: A two-tier memory system (auto-loaded + on-demand) with sources for first-party content, all organized across privacy-bounded spaces.

Comparison to traditional note-taking systems

If you're familiar with tools like Notion, Evernote, or Obsidian, here's how Rebel memory compares:

  • Notion/Evernote: Full content storage - you write entire notes. Rebel: Pointer system - you signpost to where info lives.
  • Obsidian: Flat file structure with links. Rebel: Hierarchical tiers (auto-load vs on-demand) plus space boundaries.
  • Traditional wikis: Human-centric organization. Rebel: Optimized for AI context management with utility thresholds.

Key difference: Rebel memory is designed for AI consumption first, not human browsing. It prioritizes source attribution, utility-based tiering, and privacy boundaries.

Pass 1: High-Level Overview

Let's understand what happens when you interact with Rebel and memory gets created or accessed.

The Basic Flow

  1. You start a conversation → Rebel automatically loads your Chief-of-Staff/README.md (auto-loaded context)
  2. You ask about a specific topic → Rebel searches and reads relevant memory/topics/*.md files (on-demand)
  3. You provide new information → Rebel decides where to save it based on sensitivity and utility
  4. Rebel needs source details → It fetches from external systems using signposted queries/URLs (SSOT)

Key Insight

Memory isn't just storage - it's a routing and retrieval system. The AI uses it to know where to look for information when you ask questions.

What Gets Stored Where?

At a high level:

  • Chief-of-Staff/README.md: Your identity, preferences, cross-space context
  • Chief-of-Staff/memory/topics/: Personal deep-dives, salary info, private reflections
  • work/Company/README.md: Team-wide auto-loaded facts
  • work/Company/memory/topics/: Detailed project info, client details, team documentation

Pass 2: Memory Architecture

Now let's understand each component of the memory system more deeply.

Auto-Loaded Context (README.md)

Purpose: Essential facts that should be present in 50%+ of conversations.

Location: Chief-of-Staff/README.md, work/Company/README.md

Size limit: Keep concise - this is loaded into every conversation's context window

What belongs in auto-loaded context?
  • Your name, email, timezone, location
  • Current role and teams
  • Active priorities this month
  • Frequently referenced projects or clients
  • Communication preferences (e.g., "draft emails, don't send")
  • Links to topic files for detailed context

Example auto-loaded section from Chief-of-Staff/README.md:

## Profile

Key facts:
- Name: [Your Name]
- Email: [your.email@company.com]
- Teams: [Your teams]
- Location/timezone: [Your timezone]

## AI Working Context

- Communication preferences: [e.g., "Prefers concise, actionable responses"]
- Current focus: [Active projects, priorities]
- Working on: [[topics/Current-Project]]

## Frequently Useful References

- [[topics/Project-Name]] - Brief description (25%+ utility topics only)

Note: Details live in topic files, not here. This section just signposts to detailed context.

How Rebel loads this automatically

During Rebel's startup process for every conversation:

  1. Reads root workspace AGENTS.md (symlink to rebel-system/)
  2. Follows instruction to read Chief-of-Staff/README.md
  3. Loads any space-specific README.md if you're working in a particular context
  4. All of this happens before Rebel responds to your first message

You don't need to @ mention these files - they're always present.

On-Demand Topics (memory/topics/)

Purpose: Detailed context loaded only when relevant (<50% conversation utility).

Location: Chief-of-Staff/memory/topics/, work/Company/memory/topics/

Access pattern: Rebel searches for relevant files using semantic search (@files) or you @ mention them explicitly

What belongs in on-demand topics?
  • Deep-dives on specific projects or technologies
  • Client relationship history and notes
  • Meeting patterns and recurring workflows
  • Detailed technical specifications
  • Historical decisions and their rationale
  • Signposts to original sources (Gmail queries, Notion URLs, file paths)

Example topic file structure:

---
updated: 251026
tags: [client, consulting, acmeconsulting]
sensitivity: 0.3
description: "Client relationship context for Acme Corp"
---

# Acme Corp Client Context

## Quick summary
Tech startup client since March 2024, AI enablement project, monthly retainer.

## Signposts (SSOT)
- Primary: Gmail query: `from:contact@acme.com OR to:contact@acme.com newer_than:6m`
- Contract: /Users/jane/Documents/Contracts/Acme-2024.pdf
- Project board: https://notion.so/acme-project-board

## Key facts
- Contact: Sarah Johnson (CTO)
  - SSOT: Gmail query above | Updated: 251026 | Confidence: high
- Contract value: $8k/month retainer
  - SSOT: /Users/jane/Documents/Contracts/Acme-2024.pdf | Updated: 251026 | Confidence: high

## See also
- [[topics/AcmeConsulting/services]] - Service offerings
- [[skills/client-management/meeting-prep]] - Meeting prep workflow

Note: Facts include source attribution. No email content is duplicated here.

Single file vs folder structure decision

Use single file (memory/topics/topic-name.md) when:

  • Topic is simple with <2 subtopics
  • No supporting artifacts needed
  • Content likely <50 lines

Use folder (memory/topics/topic-name/) when:

  • 2+ distinct subtopics (each >20 lines), OR
  • Supporting artifacts needed (CSVs, images), OR
  • Items benefit from individual @-mentionability

Example folder structure:

memory/topics/AcmeConsulting/
  AcmeConsulting.md              # Main overview (entry point)
  services.md               # Service catalog
  client-Acme.md            # Client-specific detail
  client-BetaCorp.md        # Another client
  invoicing-workflow.md     # Process documentation

The main AcmeConsulting.md file acts as an index signposting to the detail files.

Sources (memory/sources/)

Purpose: Capture first-party content (meetings, emails, Slack threads) with provenance metadata for traceability.

Location: memory/sources/YYYY/MM-MMM/DD/ (organized by date)

Access pattern: Topics cite sources for provenance; sources don't backlink to topics (one-way linking)

What qualifies as a source?

Any raw, original content that could be cited as evidence or detail for a topic:

  • Meetings — Transcripts from Fireflies, Otter, Fathom, Zoom (always create source files for meetings)
  • Slack/Teams threads — Substantive discussions (5+ messages, decisions made)
  • Email threads — Important negotiations, decisions, long threads
  • Notion/Confluence docs — Rich, frequently-referenced documents
  • PDFs — Reports, contracts, research papers worth referencing
  • Linear/Jira tickets — Major features, architectural decisions

When NOT to create a source file: Quick Q&A, routine updates, ephemeral chat, bug fixes, simple pages. For these, just extract relevant facts directly to topic files.

Source file structure

Filename format: yyMMdd_HHmm_source-type_description.md

Example: memory/sources/2025/12-Dec/15/251215_1000_meeting_q3-review.md

---
description: Q3 quarterly review with leadership
source_type: meeting
source_system: fireflies
source_account: user@company.com
source_uid: abc123xyz
source_url: https://app.fireflies.ai/view/abc123xyz
stored_at: 2025-12-15
occurred_at: 2025-12-15
participants: [Jane Smith, Bob Chen, Carol Davis]
duration_minutes: 45
---

# Q3 Quarterly Review

## Summary

Brief 3-5 sentence summary of key outcomes and decisions.

## Key Takeaways

- Important decision 1
- Important decision 2
- Action item assigned

## Full Content

[Complete meeting transcript with speaker attribution]

For full details, see source-capture skill.

Topics cite sources for provenance

When you add facts to a topic file, cite the source:

## Key facts

- Budget: $500k (confirmed in [[sources/2024/10-Oct/15/241015_1000_meeting_q3-review]])
- Timeline: Q1 launch (from [[sources/2024/11-Nov/03/241103_1430_email_timeline-discussion]])

One-way linking: Topics cite sources, but sources do NOT backlink to topics. This keeps source files stable when you reorganize your knowledge.

Sources vs External SSOT

Sources are first-party content you've captured and preserved in your workspace (meeting transcripts, saved emails). External SSOT (Single Source of Truth) refers to live data in external systems that you point to rather than copy — like Gmail queries that fetch current emails, or Notion URLs. Both approaches avoid duplication, but sources preserve a snapshot while SSOT pointers always fetch current data.

Pass 3: Space Boundaries and Privacy

Now that you understand the memory architecture, let's explore where to save information based on privacy requirements.

Understanding Spaces

Spaces are folders within your Rebel workspace that have different sharing settings. Think of them as security zones.

Standard Space Structure

Rebel-workspace/
  Chief-of-Staff/           🔒 Your private space
    README.md               (Your identity, preferences)
    memory/topics/          (Personal topics, sensitive info)
    memory/sources/         (Personal source captures)

  work/Company/             👥 Shared work space
    README.md               (Team can see this)
    memory/topics/          (Team-visible project context)
    memory/sources/         (Shared source captures)
    skills/                 (Shared workflows)

  rebel-system/             🌍 Platform (read-only)
    skills/
    help-for-humans/
Space types and their privacy levels

Chief-of-Staff/ (Your private space):

  • Privacy: 🔒 Private - Never shared with anyone
  • Purpose: Your personal command center for cross-space context
  • What goes here: Personal identity, preferences, compensation info, private reflections
  • Created: Automatically during Rebel onboarding

work/Company/ (Shared work space):

  • Privacy: 👥 Shared with your team/company
  • Purpose: Team skills, client context, project documentation
  • What goes here: Anything your colleagues should have access to
  • Usually connected from: Google Drive, OneDrive, or Dropbox shared folder

rebel-system/ (Platform core):

  • Privacy: 🌍 Public (read-only)
  • Purpose: Platform skills, templates, documentation maintained by Mindstone
  • Note: You cannot modify these files
Space frontmatter (how AI identifies spaces)

Each space's README.md includes YAML frontmatter to help Rebel route correctly:

---
rebel_space_description: "Company-wide skills and memory for Mindstone"
space_type: "company"        # personal | company | team | shared | project | router
sharing: "company-wide"      # private | team | company-wide | public
sensitivity: "standard"      # standard | confidential | restricted
related_spaces: ["Exec", "Chief-of-Staff"]
owner: "jane@company.com"
---

How Rebel uses this:

  • sharing field → Determines if content can be referenced in team contexts
  • sensitivity → Affects how carefully Rebel handles information
  • related_spaces → Helps Rebel know which other spaces might have relevant context

Sensitivity Markers

Even within a single space, you can mark different sections with different privacy levels.

Example: Custom Privacy Rules

You can add your own privacy rules to your Chief-of-Staff/README.md. For example:

  • "Compensation information is PERSONAL — never save salary, equity, or financial negotiations to shared memory"
  • "Health information stays private"
  • "Draft ideas stay in Chief-of-Staff until I explicitly share them"

Rebel will follow these rules when deciding where to save information.

Section-level sensitivity markers

Within any memory topic file, you can use section headers to separate content by visibility:

# Client Meeting Notes - Acme Corp

## SHAREABLE
- Meeting date: Jan 15, 2026
- Attendees: Jane (me), Sarah (Acme CTO), Tom (colleague)
- Topics discussed: Q1 roadmap, API integration timeline
- Next steps: Draft technical proposal by Jan 22

## PERSONAL
- My concerns: Timeline is aggressive, might need to push back
- Personal note: Sarah's communication style reminds me of previous client
- Mental reminder: Don't commit to deadline without checking team capacity

Rule: Only create sections when content exists for that sensitivity level. When unclear about sensitivity, ask the user - never assume something is shareable.

Section header patterns:

  • ## PERSONAL - Never leaves Chief-of-Staff/, even if file is elsewhere (salary, equity, private reflections)
  • ## SHAREABLE - Can be shared in the work space (client facts, project status)
File-level sensitivity (frontmatter)

You can also mark entire files with a sensitivity score (0.0-1.0 scale):

---
updated: 251026
tags: [compensation, personal]
sensitivity: 0.9  # High - careful handling required
description: "Salary and equity compensation details"
---

Sensitivity scale interpretation:

  • 0.0-0.3: Low - General work information (project updates, client names)
  • 0.4-0.6: Medium - Internal company info (strategy, roadmaps)
  • 0.7-1.0: High - Sensitive personal/confidential (salary, HR matters, legal)

How Rebel uses this: When sensitivity >0.7, Rebel will be extra cautious about where it references this information and will ask before sharing details.

Making Storage Decisions

When new information needs to be saved, here's a simple two-question flow:

Question 1: Is this information sensitive or personal?

Examples of sensitive/personal:

  • Salary, equity, compensation negotiations
  • Personal health information
  • Private reflections or feelings about colleagues
  • Financial details (bank info, personal expenses)

If YES → Save to Chief-of-Staff/

If NO → Continue to Question 2

Question 2: Should your team have access to this?

Examples of team-shareable work content:

  • Client relationship history (so colleagues can context-switch)
  • Project documentation (so team understands status)
  • Workflow guides and skills (so team can follow same process)

If YES → Save to work/Company/ (shared work space)

If NO → Save to Chief-of-Staff/ (your private space)

When in Doubt, Ask

Rebel will ask you before saving potentially sensitive information to shared spaces. It's always better to start more private and move outward with explicit permission than to accidentally leak sensitive info.

Every memory topic file follows a standard template to ensure consistency and proper source attribution.

Standard memory file template (explained)

tags: [topic, domain, workflow]

2-3 sentences with main takeaways. Keep details out; link to SSOT below.

- Primary SSOT: /path/to/file OR https://notion.so/... OR Gmail query: ...
- Related: Additional canonical sources
- See also: Related memory files and skills

- Fact: {short factual statement}

  - Updated: yyMMdd
  - Confidence: low | med | high

  • Frontmatter: Metadata for programmatic filtering and search

  • Signposts: Where does the canonical information live? (Critical section)
  • Key facts: Individual facts with per-fact source attribution and confidence
  • See also: Cross-references to related context (helps Rebel find connected info)

The SSOT Principle (Single Source of Truth)

The most critical principle in Rebel's memory system: Every fact must have a traceable source.

⚠️ NEVER MAKE STUFF UP

Rebel is explicitly instructed: No assumptions, guesses, or inferences in memory files. If uncertain about any detail, ask the user first. It's always better to ask than to assume.

Why SSOT matters

Problem scenario without SSOT:

  1. You tell Rebel: "Client Acme's budget is $10k/month"
  2. Rebel saves it to memory
  3. Three months later, client actually confirms $12k/month via email
  4. Memory file still says $10k (now incorrect)
  5. You ask Rebel: "What's Acme's budget?" → Gets wrong answer

Solution with SSOT:

  1. You tell Rebel: "Client mentioned $10k in our call"
  2. Rebel saves: "Budget: ~$10k (preliminary) - SSOT: Personal note from call Jan 5 | Confidence: low"
  3. Three months later, email confirms $12k
  4. Rebel updates: "Budget: $12k/month - SSOT: Gmail query: from:client@acme.com subject:contract | Confidence: high"
  5. You ask Rebel: "What's Acme's budget?" → Gets correct answer from source

Benefits:

  • Trust in memory accuracy
  • Easy verification path
  • Clear confidence levels
  • Graceful handling of uncertainty
SSOT source format examples

Gmail query format:

SSOT: Gmail query: `from:client@acme.com subject:"contract" newer_than:3m`

Rebel can execute this query to fetch current emails when needed.

Notion URL format:

SSOT: https://notion.so/workspace/page-title-abc123def456

Direct link to Notion page (Rebel can fetch via built-in Notion connector).

File path format:

SSOT: /Users/jane/Documents/Contracts/Acme-2024-03-15.pdf

Absolute path to local file (Rebel can read).

Folder location format:

SSOT: /Users/jane/Documents/ProjectX/ (see README.md and specs/)

Directory containing multiple related files.

Slack search format:

SSOT: Slack #engineering channel, search: "database migration" after:2025-12-01

Slack search query that retrieves relevant thread.

Web URL format:

SSOT: https://company.com/docs/technical-specs (retrieved 2025-12-16)

External public URL (include retrieval date since web content changes).

When to Update Memory

Not every conversation results in memory updates. Here are the guidelines Rebel follows:

✅ DO update memory when...
  • User explicitly states biographical facts: "I work at Acme Corp as Engineering Manager"
  • User expresses clear preferences: "Always draft emails for my review, never send directly"
  • User corrects Rebel multiple times: "No, the meeting is Thursday, not Wednesday" (pattern suggests memory gap)
  • User provides context to remember: "Save this - our design system uses Tailwind CSS"
  • User explicitly requests memory update: "Add this to my memory" or "Remember this"
❌ DON'T update memory when...
  • Making assumptions or inferences: User mentions "project deadline" → Don't assume which project without confirmation
  • Recording temporary session details: "I'm debugging this script today" → Too transient for memory
  • Capturing conversational back-and-forth: Don't log every message exchange, only key factual outcomes
  • Speculating about preferences: User used formal tone once → Don't assume they always prefer formal communication
  • Duplicating information from SSOT: Don't copy entire email threads, just signpost to them
Memory update process (how Rebel does it)
  1. Determine what to update: Biographical, preference, domain-specific context?
  2. Check if file exists: Read the current file first (never blindly overwrite)
  3. Make minimal, precise updates: Surgeon mode - change only what needs changing
  4. Maintain cross-references: Check if other files should link to this new information
  5. Verify with user if uncertain: When in doubt about where or what to save, ask first

Example conversation triggering memory update:

User: "I've joined the Executive team at Acme as of this month."

Rebel: "I'll update your Chief-of-Staff/README.md to reflect your new team membership. Should I also note this in work/Acme/Exec/README.md for team-visible context?"

User: "Yes, please."

Rebel: Updates both files with minimal changes:

  • Chief-of-Staff/README.md: Adds "Exec" to TEAMS list
  • work/Acme/Exec/README.md: Adds you to team member list

Cookbook: Common Scenarios

Practical examples of how to handle common memory situations.

Scenario 1: You got a raise - where to save it?

Situation: Your manager just confirmed a salary increase to $120k/year plus 1.5% equity.

Solution:

  1. Location: Chief-of-Staff/memory/topics/Compensation-YourName.md (NOT in any work space)
  2. Section: ## PERSONAL (even though file is already private)
  3. Include SSOT: Link to email from manager or contract document
