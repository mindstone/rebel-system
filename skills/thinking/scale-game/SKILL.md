# Scale Game

Test ideas by stretching them to extremes. Things that work at one scale often break at another, and discovering where they break reveals the real constraints.

## See also

- [`devils-advocate`](../devils-advocate/SKILL.md) — structured critique
- [`simplification-cascades`](../simplification-cascades/SKILL.md) — reducing complexity

## [GOAL]

Expose hidden assumptions, breaking points, and surprising strengths in an idea by imagining it at radically different scales.

## [PROCESS]

1. **Establish the baseline**
   - Ask the user to describe the idea, plan, or approach in its current form
   - Clarify: what scale does it currently operate at? (team size, revenue, user count, time horizon, geographic scope, etc.)

2. **Scale up (10x)**
   - "What if this had to serve 10x more people / handle 10x more volume / operate in 10x more markets?"
   - What breaks first? (process, people, technology, economics, culture)
   - What stays the same? (these are your durable foundations)
   - What becomes easier at scale? (network effects, economies, leverage)

3. **Scale down (1/10)**
   - "What if you had to do this with 1/10 the resources / time / people?"
   - What would you keep? (this reveals what's truly essential)
   - What would you cut? (this reveals what's actually optional)
   - Would it still be worth doing? (this tests the core value proposition)

4. **Scale time (10x faster or 10x longer)**
   - "What if the deadline was next week instead of next quarter?"
   - "What if this had to last 10 years instead of 1?"
   - Different time horizons reveal different priorities

5. **Synthesize**
   - What did the scaling reveal about hidden assumptions?
   - Which constraints are real vs self-imposed?
   - What's the insight about the right scale for this idea?
   - What should change in the current plan based on what you learned?

## [IMPORTANT]

- The point isn't to plan for 10x — it's to use 10x as a lens to see the current plan more clearly.
- Pick the most relevant scaling dimension for the situation. Don't mechanically run all four.
- Ask 1-2 questions at a time. Let the user think through each scale before moving on.
- Some ideas genuinely don't need to scale. That's a valid finding.
